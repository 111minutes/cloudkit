CloudKit via cURL

This is a tour of the CloudKit REST/HTTP 1.1 API using curl. For the complete spec, see the CloudKit REST API.

If you haven't already installed the gem:

$ gem install cloudkit

Create a rackup file named config.ru, containing these two lines of code:

require 'cloudkit'
expose :notes

Run the app using Thin:

$ thin start -R config.ru

CloudKit is discoverable from top to bottom. Let's see what resource collections we're hosting:

$ curl -i http://localhost:3000/cloudkit-meta
HTTP/1.1 200 OK
ETag: "ef2f29b1834ef8c2bf0d8f1abb100177"
Cache-Control: proxy-revalidate
Content-Type: application/json
Content-Length: 20
Connection: keep-alive

{"uris":["\/notes"]}

See what we can do with these note resources:

$ curl -i -XOPTIONS http://localhost:3000/notes
HTTP/1.1 200 OK
Content-Length: 0
Allow: GET, HEAD, POST, OPTIONS
Connection: keep-alive

List the currently available notes:

$ curl -i http://localhost:3000/notes
HTTP/1.1 200 OK
ETag: "df392c5664e6ecd64b83210fb925f6c8"
Cache-Control: proxy-revalidate
Content-Type: application/json
Content-Length: 32
Connection: keep-alive

{"uris":[],"total":0,"offset":0}

The Link header will be explained momentarily.

Create a note using POST:

$ curl -i -XPOST -d'{"title":"projects"}' http://localhost:3000/notes
HTTP/1.1 201 Created
Cache-Control: no-cache
Content-Type: application/json
Content-Length: 157
Connection: keep-alive

{
"uri":"notes\/0dda06f0-b134-012b-a2d8-0017f2c62348",
"ok":true,
"etag":"0dda0de0-b134-012b-a2d8-0017f2c62348",
"last_modified":"Sun, 21 Dec 2008 02:21:52 GMT"
}

List the currently available notes again:

$ curl -i http://localhost:3000/notes

Create a note using PUT so that we can specify its location:

$ curl -i -XPUT -d'{"title":"reminders"}' http://localhost:3000/notes/abc
HTTP/1.1 201 Created
Cache-Control: no-cache
Content-Type: application/json
Content-Length: 126
Connection: keep-alive

{
"uri":"\/notes\/abc",
"ok":true,
"etag":"89487620-b134-012b-a2d8-0017f2c62348",
"last_modified":"Sun, 21 Dec 2008 02:25:19 GMT"
}

View the new note:

$ curl -i http://localhost:3000/notes/abc
HTTP/1.1 200 OK
Last-Modified: Sun, 21 Dec 2008 02:25:19 GMT
ETag: "89487620-b134-012b-a2d8-0017f2c62348"
Link: <http://localhost:3000/notes/abc/versions>; rel="http://joncrosby.me/cloudkit/1.0/rel/versions"
Cache-Control: proxy-revalidate
Content-Type: application/json
Content-Length: 21
Connection: keep-alive

{"title":"reminders"}

Along with the usual metadata, individual resources also provide discovery information via Link Headers as shown above. These links allow user agents to find related resources such as the complete history of a document and its ETags. The utility of these items will be demonstrated momentarily.

Attempt a careless update and enjoy the failure:

$ curl -i -XPUT -d'{"title":"foo"}' http://localhost:3000/notes/abc
HTTP/1.1 400 Bad Request
Cache-Control: no-cache
Content-Type: application/json
Content-Length: 25
Connection: keep-alive

{"error":"etag required"}

Succeed in updating by being specific:

$ curl -i -XPUT -H'If-Match:89487620-b134-012b-a2d8-0017f2c62348' -d'{"title":"foo"}' http://localhost:3000/notes/abc
HTTP/1.1 200 OK
Cache-Control: no-cache
Content-Type: application/json
Content-Length: 126
Connection: keep-alive

{
"uri":"\/notes\/abc",
"ok":true,
"etag":"522be9f0-b135-012b-a2d8-0017f2c62348",
"last_modified":"Sun, 21 Dec 2008 02:30:56 GMT"
}

(Note: Your ETag will likely be different so substitute the one that curl provided when you created your own "abc" resource.)

Watch a secondary, out-of-date client fail at updating by being specific but also being stale:

$ curl -i -XPUT -H'If-Match:89487620-b134-012b-a2d8-0017f2c62348' -d'{"title":"foo"}' http://localhost:3000/notes/abc
HTTP/1.1 412 Precondition Failed
Cache-Control: no-cache
Content-Type: application/json
Content-Length: 31
Connection: keep-alive

{"error":"precondition failed"}

A 412 code is returned indicating a precondition for this request failed. Specifically, the ETag was out of date. In this case, our second client can fall back on the resource's history to "catch up" and apply its changes to the most recent version of the resource.

Get all versions of the document using the URI tagged with rel="versions" in the link header mentioned above, reverse sorted by Last-Modified, feed style:

$ curl -i http://localhost:3000/notes/abc/versions
HTTP/1.1 200 OK
Last-Modified: Sun, 21 Dec 2008 02:30:56 GMT
ETag: "bd9226d0c5f34fd761f6c64a0fe8bcc9"
Cache-Control: proxy-revalidate
Content-Type: application/json
Content-Length: 109
Connection: keep-alive

{
"uris":[
"\/notes\/abc",
"\/notes\/abc\/versions\/89487620-b134-012b-a2d8-0017f2c62348"
],
"total":2,
"offset":0
}

Delete the document:

$ curl -i -XDELETE -H'If-Match:522be9f0-b135-012b-a2d8-0017f2c62348' http://localhost:3000/notes/abc
HTTP/1.1 200 OK
Cache-Control: no-cache
Content-Type: application/json
Content-Length: 174
Connection: keep-alive

{
"uri":"\/notes\/abc\/versions\/522be9f0-b135-012b-a2d8-0017f2c62348",
"ok":true,
"etag":"522be9f0-b135-012b-a2d8-0017f2c62348",
"last_modified":"Sun, 21 Dec 2008 02:30:56 GMT"
}

Try to GET it again and notice the helpful 410:

$ curl -i http://localhost:3000/notes/abc
HTTP/1.1 410 Gone
Link: <http://localhost:3000/notes/abc/versions>; rel="http://joncrosby.me/cloudkit/1.0/rel/versions"
Cache-Control: no-cache
Content-Type: application/json
Content-Length: 37
Connection: keep-alive

{"error":"entity previously deleted"}
